# AGENTS.md

> 이 문서는 momso 팀(사람 + AI 에이전트)이 사설 작업 레포에서 따르는 규약의 **공개판**이다.
> 원칙과 규칙 본문은 사설 정본 그대로이며, 사설 레포 내부 경로·운영 문서를 가리키는 절(규칙 지도·문서 배치 규칙 등)만 정리했다.
> AGENTS.md는 Linux Foundation(Agentic AI Foundation) 오픈 표준 형식이다 — https://agents.md

## 협업 원칙

- 설명은 쉽게 푼다. 계획을 세우고 진행 상황을 기록하며 피드백 루프를 유지한다.
- `momso` 공식 표기는 항상 소문자 `momso`다.
- UI/UX 수정은 사용자의 명시적 지시 없이 임의로 확장하지 않는다.

## 충실도 가드 — 요약·정리·변환이 '결정'을 왜곡하지 않게 (최우선, 어기지 말 것)

AI가 raw(전사·통화·원문)나 기존 문서에서 **회의록 정리본·리서치 보고서·요약·공개 변환**을 만들 때, 일반화·패러프레이즈가 *결정의 성격*을 조용히 바꿀 수 있다. 실제 사고 사례: "A가 B에게 PoC 전사 **파트너로 제안**"이 "B **벤더 사용/평가**"로 뭉개짐. 이런 왜곡이 누적되면 결정 기록 자체를 못 믿게 된다.

1. **결정은 일반화 금지.** 일반화·패러프레이즈는 *배경·설명*에만 허용한다. **결정·의향·제안·합의·거절·보류·약속·수치는 원본의 구체(누가 · 누구에게 · 무엇을 · 어떤 조건으로)를 그대로 보존**한다. 동사(제안/합의/거절/보류)와 조건절(…하면, …통과 시)을 바꾸지 않는다.
2. **원본 보존 + 추적가능.** raw/원천은 요약으로 대체하지 않고 항상 남긴다. 요약·정리본의 각 핵심 주장은 *원본 어느 대목에서 왔는지* 대조 가능해야 한다(가능하면 출처 표기).
3. **충실도 검사(생성 후 필수).** 요약·정리본·보고서·공개 변환을 만든 뒤, 거기서 **결정·의향·합의·수치를 모두 뽑아 원본과 1:1 대조**한다. 의미가 바뀐 곳(동사·조건·주체·수치 변형, 원본에 없던 단정 추가)을 **'충실도 로그'로 남기고 사람(검수자)에게 보인다.** 하나라도 의미 변형이 있으면 **사람 확인 전 확정·발행 금지.**

> 몸소 제품의 핵심(AI 초안 → 강사 검수)을 우리 레포 운영에 그대로 적용한 것. 요약이 결정을 왜곡하면 반드시 플래그가 떠서 사람이 잡는다.

## Git 안전 규칙

- `git status --short`로 현재 변경사항을 먼저 확인한다.
- 공동작업자의 로컬 변경사항을 덮어쓰지 않는다.
- pull이 필요하고 워킹트리가 깨끗하면 `git pull --ff-only`를 우선한다.
- 로컬 변경사항이 있으면 임의로 stash/reset/checkout하지 말고, 사용자에게 현재 상태와 선택지를 설명한다.

## 브랜치 · PR · 배포 전략 (모노레포 — 다음 세션·AI 필독, 어기지 말 것)

이 레포는 한 모노레포에서 여러 앱(`apps/alpha` 알파 · `apps/inbodylike-demo` 데모 · `apps/web` 베타(예정) · `apps/mobile`(예정))이 자란다. 세션마다 제각각 브랜치를 쓰면 트렁크가 껍데기가 된다(실제로 그렇게 됐다). 아래를 지킨다.

### 실제 협업 모델 (누가·어떻게 일하나 — 2026-07-01 확정, 규칙은 여기에 맞춘다)
이 레포는 "여러 개발자의 대칭 협업"이 아니라 아래 **비대칭 구조**다.
- **김성균 = 유일한 writer.** 코드·문서·회의록 업로드는 사실상 전부 김성균이 한다. 시간이 없어 **여러 AI 세션을 동시에** 돌린다. → **브랜치·PR이 겹치는 진짜 원인은 두 사람의 충돌이 아니라 이 병렬 세션**이다.
- **유동환 = read-only 소비자.** 최신본을 받아 진행 상황을 확인하는 용도이며, 레포에 직접 업로드하는 일은 드물다.
- 규칙을 지금부터 대규모 팀용으로 무겁게 만들지 않는다.

### 병렬 세션 규율 (동시에 여러 세션을 돌리므로 필수)
- **한 세션 = 한 브랜치.** 두 세션이 같은 브랜치를 만지지 않는다.
- **동시에 여럿이면 세션마다 자기 git worktree.** worktree = 같은 `.git`을 공유하되 작업 폴더만 분리 → 세션 A(알파)와 세션 B(문서)가 파일 충돌 없이 병렬 진행. 끝나면 `git worktree remove`로 정리.
- **끝나면 즉시 PR → 머지 → 브랜치(및 worktree) 삭제.** 안 닫고 다음 세션을 열면 우후죽순이 된다 — 닫는 것까지가 한 세션이다.
- **'작업 완료'의 정의 = PR 머지 + 브랜치 삭제까지.** 코드·문서가 됐다고 끝이 아니다. AI는 작업이 완결되면 **잊지 말고 PR을 열고 머지까지** 챙기고 세션을 닫는다. (`delete_branch_on_merge` 켜둠 → 머지되면 브랜치는 자동 삭제.)

### 트렁크 = `main`
- `main`이 **단일 통합 트렁크.** 모든 앱·문서가 여기로 모이고 **배포는 main에서** 나간다. main은 항상 배포 가능 상태를 유지한다.
- ✅ **트렁크화 완료(2026-06):** 실제 앱들이 main에 있고 alpha는 **main에서 자동 배포**. 이제부터 트렁크 노릇하는 장수 브랜치를 새로 만들지 않는다.

### 작업 브랜치 (짧게 살고 PR로 병합)
- 모든 작업은 **main에서 딴 짧은 브랜치**에서. 끝나면 **PR → main → 브랜치 삭제.**
- "짧은"의 정량 기준(DORA): 가능하면 **활성 브랜치 ≤3개 · 브랜치 수명 수시간~며칠 · 자주 병합.** 오래 끌수록 트렁크에서 멀어져 충돌·리스크가 커진다.
- 네이밍 `<type>/<scope>-<슬러그>` — type: feat|fix|chore|docs|research|refactor / scope: 앱·영역(alpha·beta·mobile·demo·brand·infra 등). 예: `feat/alpha-byo-routing`, `fix/alpha-workspace-switch`.
- **한 브랜치 = 한 관심사.** 앱 코드·경쟁 조사 문서·회의록을 한 브랜치에 섞지 않는다.

### `main` 직접 커밋·푸시 금지 (AI 포함)
- 항상 브랜치 → PR. 병합은 김성균 검토 후(사소한 docs는 셀프 머지 가능하나 명시).
- **`git add -A` 금지** — 변경 경로를 명시적으로 add(다른 세션 WIP·타 관심사 안 쓸려가게). 커밋 전 `git status --short`로 관심사 밖 더티 확인.

### 커밋 메시지
- Conventional Commits: `feat(alpha): …` `fix(alpha): …` `docs(brand): …`. 한글 브리핑체 OK. 끝에 `Co-Authored-By: <모델>` 트레일러.

### 배포 (Vercel) — 앱별 프로젝트 + Root Directory
- 각 앱 = 자기 Vercel 프로젝트, **Root Directory = 그 앱 서브디렉터리**, **사설 작업 레포에 git 연결**, **그 서브디렉터리 변경 시에만 빌드**, **프로덕션 브랜치 = main**.
  - momso-alpha → `apps/alpha` · momso-demo → `apps/inbodylike-demo` · (베타·모바일 동일 패턴).
- **시크릿은 코드에 절대 두지 않는다 → Vercel 환경변수.** (`.env*.local`은 gitignore. Local/Preview/Production 환경별 분리.)
- **PR마다 Vercel Preview 배포가 자동 생성**된다 → 병합 전 실제 배포본으로 검증한다(production=main만 보지 말 것).
- **이 공개 레포(mow-coding/momso)는 배포 소스가 아니다** — 마스킹·검수된 전시/prior-art 사본. 민감자료(raw 전사·개인정보 등)는 공개하지 않으며, 공개/마스킹 분류 기준은 사설 레포에서 관리한다.
- 수동 CLI 배포는 핫픽스/프리뷰 한정 + **반드시 `.vercel/project.json` 링크 확인 후** 실행(unlinked `vercel --prod`는 디렉터리명 프로젝트를 자동 생성하는 사고가 실제로 났다).

### 공유 인프라 보호
- Vercel 프로젝트·도메인 리네임/삭제, git 연결 변경, 레포 public 전환은 **명시적 지시 없이 하지 않는다.** `momso.vercel.app`은 향후 정식 제품용 **예약 도메인**(어느 프로젝트에도 붙이지 말 것).

### AI 세션 시작 체크리스트
1. 이 문서 읽기 → 2. `git status --short` + 현재 브랜치 확인 → 3. 작업용 브랜치 만들기(main에서 분기, main 직접작업 금지 — 다른 세션과 **동시 작업이면 별도 worktree**) → 4. 한 관심사만 → 5. PR로 마무리(머지 후 브랜치·worktree 삭제) → 6. 공유 인프라는 함부로 안 건드림.

### 규칙은 문서가 아니라 GitHub로 강제한다 (enforcement)
문서 규칙만으로는 안 지켜진다(특히 AI 세션). 업계 표준은 **GitHub 브랜치 보호(branch protection / rulesets)**로 기술적으로 막는 것:
- main에 **PR 필수 + 승인 ≥1 + 직접 push 차단**(사람도 AI도 main에 직접 못 올림). `CODEOWNERS`는 단독으로는 리뷰어 "제안"만 하므로 보호 규칙과 함께 써야 강제된다.
- 새 커밋 push 시 **stale 승인 자동 해제**, 가능하면 **필수 상태체크(CI)** + **Conventional Commit 형식 검사**(ruleset metadata).
- ⚠️ **아직 미설정(2026-07-01).** 무료 사설 레포라 브랜치 보호(rulesets)가 제한될 수 있으니 설정 가능 여부부터 확인한다. 안 켜지면 그때까진 **로컬 pre-push 훅으로 main 직접 push 차단**(이 훅이 원격 main 삭제를 실제로 막아준 적 있다 — 유지).

### 모노레포 빌드 스킵 — 정확한 메커니즘
"서브디렉터리 변경 시만 빌드"는 Vercel **Skip unaffected projects**(GitHub 연결 + JS 워크스페이스 필요, 빌드 슬롯 미점유)를 쓰는 게 표준. **워크스페이스(고유 `name` + 내부 의존 명시) 미구성이면 변경이 "전역"으로 잡혀 전체 앱이 배포된다.** 워크스페이스를 구성하거나, 그게 어려우면 Ignored Build Step 스크립트로 대체.
