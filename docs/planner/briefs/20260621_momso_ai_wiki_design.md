## 0. 전제 (모든 선택의 근거)

- **규모**: 단일 요가원·강사 1~3명(알파=유동환 1명). 알파 ~1,000청크, 단일 요가원 1년 ~7,500청크. 100개 요가원이어도 ~75만 청크 — **단일 Postgres+pgvector가 가볍게 감당하는 소규모**. 과설계(전용 벡터DB·LangChain) 금지.
- **도메인**: 요가 지식은 체화·암묵지 → **원본(objet)이 1차 권위층**. 가치 단위 = "질문 정답"이 아니라 "강사↔수련생 사이 시간에 걸친 맥락" → **종단 회상이 1급**. 금기·부상 자세 오권유 = 신체 위해 → **안전 recall > precision**.
- **한국어**: 구어체 전사·교착어·산스크리트/영어/한글 혼용 → **임베딩 한국어 품질이 1급 선택 기준**.
- **프로바이더 미정**: 생성·임베딩 모두 어댑터 뒤로 격리. 프로바이더는 설계의 *전제가 아니라 출력*.

## 2. 프로바이더 선택 (출력 — 어댑터 뒤, 교체 가능)

**① 임베딩 (한국어 품질만, 생성과 독립):**
- 알파: **BGE-m3** (DeepInfra 매니지드, **1024차원**, $0.01/1M) — 즉시 가용.
- 승급: **KURE-v1** (고려대, Ko-MTEB 리트리벌 1위 0.5264, MIT, 1024차원 동일) — 한국어 recall이 병목이면 `baseURL`만 교체 + 소량 재색인. *요가 한국어 평가셋 검증 후* 결정.
- OpenAI text-embedding-3 기각(한국어 열위 0.4488 + 락인).
- 스키마: `vector(1536)→vector(1024)` + `model_id`·`dim` 메타컬럼 + "단일 활성 모델" 규약.

**② 생성 (현 점수 1위, 단 미정·어댑터 뒤):**
- **Claude Opus 4.8** 채택(tool-use·structured output·streaming 우위; 5/25 $/MTok). GPT-5.5·Gemini 3.1 Pro는 어댑터 뒤 후보.
- 단일 모델 고정으로 알파 시작(라우팅 유보 — 모델 교체가 캐시 깸·품질회귀 은닉). `thinking={type:"adaptive"}`, effort high 기본.
- ※ 프로바이더 미정 원칙 유지 — env 한 줄로 교체. 임베딩(B)·생성(A) 혼합 정상.

**격리 인터페이스 (SDK 타입 import 금지, CI 강제):**
```ts
interface EmbeddingProvider  { id; dim; maxTokens; embed(texts): Promise<number[][]> }
interface GenerationProvider { id; generate(req): AsyncIterable<TokenChunk> }
```

## 3. ★ 무결성 게이트 (= "진실성=기술적 무결성"의 기술 실현) ★

**1순위 불변식. "AI는 추론에선 틀려도, 귀속·인용에선 절대 안 틀린다."** (실제로 AI가 회의록 화자 매핑을 틀린 전례 있음 → momso에선 치명적 비전 실패.)

| 강제 | 메커니즘 |
|---|---|
| 화자 라벨 출처 보존 | diarization은 전사 벤더 산출을 발화 단위 **불변 저장**, LLM 재배정 금지. |
| 화자 **정체** 매핑 | 딕셔너리가 *제안* → **강사가 확인**한 것만 유효(자동 단정 금지). diarization(A/B 분리)과 정체(A=유동환)를 구분. |
| 인용 = 원본 스팬 고정 | 뒷받침 발화 스팬 없으면 그 주장 자체를 못 함. 패러프레이즈를 사실로 단정 금지. |
| '모름' 강제 | 화자/근거 불확실 시 "불명" 표기 허용·요구. 자신 있는 추측 금지. |
| 검증 패스 | AI 초안의 모든 귀속·인용이 실제 스팬+일치 화자 라벨에 매핑되는지 기계 재검증 → 사람이 AI 매핑을 무심코 승인하는 2차 사고 방지. |
| `source_ref` 강제 | structured output이 모든 주장에 ref 달게. ref 없으면 발행 불가. |
| **검수 UI 가시화** | 대화 초점 항목마다 **① 화자 라벨 ② 출처 스팬(클릭→보정본 발화)** 표시 — 강사가 귀속을 *검증·교정*하는 지점(교정은 보정본·딕셔너리에 피드백). 화자는 초점의 **분류**도 결정(수련생 발화=안전신호·피드백 vs 강사 발화=큐·지도) → 화자 없으면 분류 자체가 틀림. |

## 4. ★ 보정본 provenance 파이프라인 ★

```
음성원본(objet, L0)              ← 불변 보존
  → STT 전사본(objet, L1)         ← 벤더(삼성/Tiro/Naver Cloud STT, 미정). diarization 有, 정체 無
  → 보정본(L2)                    ← 몸소 딕셔너리로 [용어 보정(자동 OK) + 화자 정체 매핑(딕셔너리 제안→강사 확인)]
                                     ★ 검수 게이트 = 보정본은 "검수된 산출물"
  → 리포트(zet)                   ← 무조건 L2(보정본) 기반
```
- **리포트 작업은 무조건 보정본 기반.** raw(L0/L1)는 불변 보존. AI는 검증용으로 raw까지 *볼 수* 있음("AI는 다 본다") — 단 보정본이 뒷받침 안 하는 건 단정 못 함.
- **보정 추적**: 보정본의 모든 수정(화자 X→Y, 용어 a→b, 누가·언제)을 기록 → 오류 시 STT/보정/리포트 중 어디서 깨졌는지 감사.
- born-digital(철학문서·CSV 등)은 STT 없음 → 가벼운 용어 정규화만.

## 5. ★ 몸소 딕셔너리 (AI의 내부 공책) ★

objet(raw)도 zet(사람 리포트)도 아닌 **제3의 참조층.** 전사본→보정본 변환에 필요한 정보만 따로 보관.
- **구성**: ① 용어 glossary(요가 용어·산스크리트 음차 통일·금기 어휘) ② 화자 정체/별칭 레지스트리(목소리·문맥 패턴 → 사람 정체).
- **성격**: AI Wiki에서 *파생·제안*되지만 큐레이션됨. 화자 정체 항목은 **사람 확인된 것만 유효**(무결성). 용어 항목은 자동 비중 높여도 됨.
- **구현**: 설계서의 `entity` / `entity_alias` 테이블 + 용어 사전 = 이 딕셔너리. owner별·버전드.
- **재사용**: 한 딕셔너리가 *보정(L1→L2) · 검색 facet · 안전 금기 recall*에 모두 쓰임 (별도 "facet 표준어휘 사전"과 동일 자산).

## 7. 길A · 길B 호출 순서

**길A 수업발**: transcript 업로드 → ingest(보정본 생성: 정규화+딕셔너리 보정, 화자정체 강사확인) → 색인 → generate(2단계: 추론→structured) → zet `draft` → 강사 검수 → `published`(北極星).

**길B 대화발**: 질의 embed → `momso_search(grounding, student_ref?)`(owner선필터 KNN+RRF, 종단부스트·supersedes해소·금기동반) → compose(사람 인용은 published만) → generate(stream, 인용 zet_id 부착) → 인용 사후가드 → 발행→검수→published → `kpi_event(revisit)`.

## 9.5 워크스페이스 (확정 — 알파 포함)

강사는 한 요가원에만 매이지 않음(여러 요가원 근무 가능) → **요가원/스튜디오 단위 워크스페이스**로 지식·수련생·위키를 분리.
- **경계 3종 동시 해결**: ① 조직(요가원별 분리) ② 기밀(수련생 안 섞임) ③ **RAG 스코프**(AI는 활성 워크스페이스 지식만 grounding — 무결성·프라이버시 boundary). + "특정 요가원 하드코딩 금지" 원칙 강화.
- **스키마**: `workspace` 테이블 + `objet·zet·student·embedding(·zet_link via zet)`에 `workspace_id`. → **빌드 1번 스키마 마이그레이션에 포함.** RLS = owner + workspace 스코프. `momso_search`도 workspace_id 필터.
- **UI**: 사이드바 최상단 워크스페이스 전환기(Notion/Slack식) → 활성 워크스페이스가 모든 탭(업로드·검수·대화·위키)을 스코프.
- **온보딩**: Supabase·키 연결 → 워크스페이스 생성/선택(예: "빅블루 요가").
- **알파 범위(정직)**: 단일 사용자(유동환)의 *여러 요가원* = 한 Supabase 내 파티션. **여러 강사가 한 워크스페이스 공유 = 베타**(BYO-Supabase와 충돌).

## 10. 알파 배포·소유 모델 (확정: BYO + 서버리스 BYOK 프록시)

> 알파 테스터 = 유동환 1명. "우리 제공 vs 사용자 제공"으로 가르되, **LLM 키는 팀이 선행 프로젝트에서 검증한 HttpOnly-쿠키 패턴**을 그대로 채택.

- **우리(momso) 제공**:
  - GitHub: 알파 코드 `momso-private/apps/alpha`.
  - Vercel: `momso-alpha` 프로젝트(별도 신규; 기존 `momso` 링크는 인바디라이크용). **정적 SPA + 얇은 서버리스 함수**(BYOK LLM 프록시). 서버는 키를 *저장하지 않는* stateless 프록시.
- **사용자 제공(BYO)**:
  - **Supabase**: 사용자가 자기 프로젝트 생성 → URL+anon키 입력. **브라우저 ↔ Supabase 직접**(anon키 공개 전제 + RLS → *진짜* client-side, 안전).
  - **LLM 키**: 사용자가 자기 OpenAI 키 입력 → **서버 함수가 AES-256-GCM 암호화 → HttpOnly 쿠키(24h)**. LLM 호출은 **서버사이드**에서 *호출 직전에만 복호화*(상태확인 시 복호화 금지). **키가 브라우저 JS에 절대 안 닿음**(DB·localStorage·sessionStorage 미저장). 가드레일: 선행 프로젝트의 BYOK 아키텍처 가드레일 문서를 그대로 따른다.
- **알파 LLM = OpenAI 단독**: 1키로 생성(GPT) + 임베딩(text-embedding-3, `dimensions=1024` → vector(1024) 유지, KURE 스왑 시 무차원변경).
- **무결성 보너스**: LLM·RAG 오케스트레이션이 서버 함수에서 도니, 인용·귀속 검증 같은 무결성 가드도 **서버사이드로** 둘 수 있음(브라우저 JS보다 견고 — §6 JS가드 변조 우려 해소).
- **아키텍처 함의(결정 필요)**: 현재 alpha는 Vite 정적 SPA → 서버 함수 자리 없음. 택1 — **(a)** Vite + Vercel `/api` 서버리스 함수(적은 변경), **(b)** Next.js 전환(Rorschach BYOK 코드 재사용·검증된 보안 이식). lean=(b).
- **베타 차이**: 이 패턴은 멀티유저에도 유효(각자 HttpOnly 쿠키). 베타 전환은 *키 노출* 문제가 아니라 결제/공용키·레이트리밋 문제.
- **온보딩 자동화(사용자 DB 셋업 부담 0)**: Supabase **"for Platforms"** 패턴 — momso=published Supabase **OAuth 앱** → 사용자 승인(짧은 토큰) → **Management API**로 ①`POST /v1/projects` 생성 ②스키마 SQL 주입(전용 `/database/migrations`는 폼 신청제 → 일반 `/database/query`로 우회) ③URL/anon키 취득. **알파=반자동**(사용자가 프로젝트만 생성, momso가 스키마 자동 주입), **베타=풀 OAuth 자동화**. 함정: 프로비저닝 ~1분·무료티어 프로젝트 수 제한·OAuth 토큰은 셋업 때만 사용.
- **임베딩(알파)**: OpenAI `text-embedding-3` (dimensions=1024) — 사용자 BYOK 키로 호출(추가 서비스 0). KURE/BGE-m3(한국어 우위)는 *호스팅 필요*라 한국어 recall이 실측 병목일 때 스왑(어댑터·1024차원이라 무변경).
- **BYOK 키 저장 = 사용자 Supabase 암호화 저장 (확정)**: momso는 지속 세션이라 휘발 쿠키(Rorschach식) 대신 *사용자 Supabase에 암호화 저장*(영속·재입력 불필요). 규칙: **평문 금지 → 암호화 저장, Vercel 함수가 서버에서만 복호화, 브라우저에 키 미노출.**
- **스키마 업데이트 전파 (사용자 버튼 한 번)**: 프로비저닝과 동일 엔진 — **버전 매겨진 마이그레이션** + 각 사용자 DB가 자기 버전 기록 → 앱이 "버전<최신" 감지 → "업데이트" 버튼 → Management API로 *밀린 마이그만* 순서 실행. 조건: ①그 순간 DDL 권한(OAuth Management 토큰) ②**전진형·데이터 안전 마이그**(파괴 변경은 사용자 실데이터 위험 → 신중) ③실패 시 롤백(트랜잭션). 분산된 사용자 DB에 안전 마이그 짜는 규율 필요.

> ⚠️ 정정: 이전 초안의 "브라우저가 OpenAI 직접 호출 → 키 노출 → 단일 사용자 한정" 카베앗은 **철회**. 위 HttpOnly-쿠키 서버 프록시 패턴이 그 문제를 애초에 제거함(선행 프로젝트에서 검증).

## 10.1 확정 정정 (2026-06-22): 2-티어 Supabase + NCP + 구글 단독 로그인

위 §10을 아래로 갱신(실제 구축 반영).

**소유 경계**
- **우리(momso) 서버사이드:** GitHub + Vercel(정적앱 + BYOK 프록시) + **momso 중앙 Supabase**(= `momso-alpha`, Seoul 리전). 중앙 Supabase는 **계정 + 기본 딕셔너리(몸소 제공 요가 용어)**만.
- **고객 BYOK = 3개:** ① 자기 Supabase(자기 데이터) ② **NCP 오브젝트 스토리지**(음성·PDF 등 원본 파일; S3 호환 어댑터, Supabase Storage 아님) ③ LLM 키.

**인증(알파)**
- **구글 단독 로그인**(GIS ID토큰 → `signInWithIdToken`). id/password 만들지 않음(베타가 휴대폰 본인인증이라 throwaway). 계정은 momso 중앙 Supabase 저장(가입 시 `profile` 자동 생성 트리거).
- **네이버 = 온디맨드:** 사용자가 원본 파일 보존(NCP)을 원할 때만 연결(NCP=Naver Cloud라 네이버 계정 필요).
- **베타+ = 휴대폰 본인인증 필수.**

**데이터 플레인 — 정정(임의 스코프 축소 철회):**
- **BYO 데이터 분리(고객 자기 Supabase에 자기 데이터)는 알파다**(§9.5·§10). 이전에 "베타"로 미룬 건 잘못 — **철회.** 지금 momso-alpha에 objet/zet를 함께 둔 건 *임시 개발 부트스트랩*(현 구현 상태)일 뿐 스코프 결정이 아니다. 알파 목표 = 고객 자기 Supabase에 데이터 + 중앙(auth/딕셔너리)/고객(데이터) 2-티어 접근(JWT 신뢰 또는 서버 프록시) 구축. → **남은 알파 작업.**
- **타입드 온톨로지 그래프도 알파다**(판정 ③ + 빌드 1번: zet_link 타입드 확장·수동 연결·owner RLS; *자동* 후보 엣지만 알파 OFF). 현재 순수 vector top-k RAG는 설계가 "부족하다"고 콕 집은 지름길 → 그래프 회수(vector + 엣지 traversal) + 관계 뷰로 보강. → **남은 알파 작업.**
- ⚠️ **작업범위는 사람(김성균/유동환)이 정한다. AI 임의 축소 금지.**

**스토리지 용어:** 오브젝트 스토리지=파일 저장 *서비스*(NCP) / 버킷=그 안의 *컨테이너* / DB(Postgres)=메타·관계·임베딩(파일은 *주소만* 저장).
