## 0. 제품 목표

### 목표

Body Note 후보 제품은 다음 4가지를 목표로 삼았었다.

1. **7개 레이어를 하나의 타임라인 위에서 동기화된 3D 뷰어로 재생**
2. **실제 해부학(A/B 등급)과 전통 이론(C 등급)을 UI에서 명확히 구분**
3. **동작별 쓰레드 기반 메모/주석 작성 및 공유**
4. **웹 브라우저 환경에서 스크러빙 가능한 성능 확보**

### 핵심 사용자

* 요가, 필라테스, 웨이트 트레이닝 지도자
* 해부학/운동학 기반 교육자
* 임상/교육 기록을 남기고 공유해야 하는 전문가

### 비목표

다음은 MVP 범위에서 제외한다.

* 진단/치료 추천 엔진
* 자동 의학적 결론 도출
* 전통 이론 레이어의 생리학적 “활성” 판정
* Full physics/FEM 수준의 물리 시뮬레이션
* 실시간 동시 편집(collaborative multiplayer editing)

## 2. 7-Layer Architecture & Authority Logic

## 2.1 레이어 정의

Body Note 후보 제품의 7개 레이어는 아래처럼 분리한다.

### Dynamic Physical Layers

1. **skeleton**

   * 뼈, 관절, 기준 좌표계
   * 입력: 관절 각도 `q`
   * 출력: 뼈 transform, 관절 pose

2. **muscle**

   * 근육 mesh
   * 입력: 관절 각도 `q`, 근수축률 `alpha`
   * 출력: skinning + corrective deformation

3. **ligament**

   * 인대 mesh 또는 ribbon proxy
   * 입력: 관절 각도 `q`, 근육 변화 proxy
   * 출력: `proxy_tension`

4. **nerve**

   * 신경 curve/mesh
   * 입력: 관절 각도 `q`, 근육 부피 변화 proxy
   * 출력: `proxy_tension`, `compression_proxy`

### Static Mapping Layers

5. **meridian**

   * 경락 curve overlay
   * 물리 연산 없음
   * 3D 좌표 기반 오버레이

6. **acupoint**

   * 경혈 point overlay
   * 물리 연산 없음
   * anchor 기반 위치

7. **myofascial_line**

   * 근막 경선 overlay
   * 물리 연산 없음
   * 교육용 라인 오버레이

> **중요:** MVP의 7번째 레이어는 `fascia tissue`가 아니라 `myofascial_line`이다.
> 실제 근막 조직의 동적 물리 시뮬레이션은 MVP 범위 밖이다.

---

## 2.2 데이터 권위(Authority) 규칙

모든 시각 요소는 아래 3등급 중 하나를 반드시 가져야 한다.

* **A = Anatomical / Verified**

  * 예: 골격, 근육의 기본 구조
  * UI 배지: `A`
  * 문구: `Anatomical`

* **B = Model-derived Proxy**

  * 예: 인대/신경의 추정 긴장도
  * UI 배지: `B`
  * 문구: `Estimated Proxy`

* **C = Conceptual / Traditional Overlay**

  * 예: 경락, 경혈, 근막 경선
  * UI 배지: `C`
  * 문구: `Traditional Overlay`

### 절대 규칙

1. **C 레이어는 A/B 레이어를 역으로 구동할 수 없다.**
2. **B 레이어는 “실측값”처럼 표기하면 안 된다.**
3. **C 레이어는 “활성화(activation)” 대신 “오버레이(overlay)”로 표기한다.**
4. **같은 화면에서 A/B/C 레이어가 함께 보일 경우, 범례와 배지가 항상 노출되어야 한다.**
5. **사용자가 C 레이어만 켠 상태에서도, A/B와 혼동되지 않도록 시각 스타일을 별도로 유지해야 한다.**

## 2.3 런타임 추론 엔진 범위

### 입력

* `q`: 관절 각도 벡터
* `alpha`: 근수축률 벡터
* `u`: 선택적 외부 부하 scalar

### 출력

* `bone_transforms`
* `muscle_deform_coeffs`
* `ligament_proxy_tension`
* `nerve_proxy_tension`
* `compression_proxy`

### MVP에서 허용되는 근사 모델

* **근육**: skinning + corrective morph + shader coefficient
* **인대**: slack/strain proxy
* **신경**: path-length delta + compression proxy
* **경락/경혈/근막 경선**: 위치 오버레이만 허용

### 금지

* Full mesh-mesh collision
* Full FEM
* 경락/경혈의 자동 생리 활성 판정
* C 레이어 기반 물리 추론

---

### [구현 핵심 로직]

```ts
// Pseudo flow
input(frame) -> resolve q, alpha
q, alpha -> worker computes:
  boneTransforms
  muscleCoefficients
  ligamentProxyTension
  nerveProxyTension

GPU renderer:
  skeleton = pose bones
  muscle = skinned mesh + corrective shader
  ligament = color/intensity/thickness by proxy_tension
  nerve = color/intensity/path offset by proxy_tension
  meridian/acupoint/myofascial_line = static overlay only
```

### 인대 프록시 예시

```ts
ligament_proxy = clamp01(
  wAngle * normalized_joint_angle +
  wAlpha * related_muscle_activation +
  wVel   * angular_velocity
)
```

### 신경 프록시 예시

```ts
nerve_proxy = clamp01(
  wPath * path_length_delta +
  wComp * compression_proxy +
  wCurve * curvature_proxy
)
```

## 3. UI/UX : Interactive Annotation System

## 3.1 워크스페이스 구조

MVP의 정보 구조는 다음 순서를 따른다.

`Workspace -> Project -> Action Instance -> Thread -> Entries / Comments`

### 정의

* **Workspace**: 사용자/팀 단위 공간
* **Project**: 수업/임상/주제 단위 묶음
* **Action Instance**: 특정 운동 데이터
* **Thread**: 하나의 주석 흐름
* **Entry**: 실제 텍스트/음성 노트 본문

---

## 3.2 Thread-based Workspace

각 `Action Instance`에는 여러 개의 독립적인 `Thread`를 가질 수 있다.

### Thread의 핵심 속성

* 하나의 `target_entity_id`를 가진다.
* 필요 시 `related_entity_ids`를 추가로 가진다.
* 하나의 `anchor_id`를 기준으로 3D 상에 연결된다.
* 하나의 `display_track`을 가진다.
* 여러 개의 `entries`를 가진다.
* 공유/복제 단위가 될 수 있다.

### Thread가 해결해야 하는 문제

* 동작별 설명을 따로 분리
* 메모를 시간축에 따라 노출
* 카메라, 강조, 설명을 함께 묶음
* 추후 공유/복제 가능

---

## 3.3 Dual-mode Memo (Adobe Style)

### Global Memo

* 전체 타임라인에서 계속 보이는 메모
* 런타임에서는 `frame 0 visible=true`인 display track으로 정규화 가능

### Keyframed Memo

* 특정 프레임/구간에서만 보이는 메모
* `opacity`, `visible`, `camera_view_id`, `active_entry_id`를 키프레임으로 제어

### 규칙

* `opacity`는 숫자 보간 가능
* `visible`, `camera_view_id`, `active_entry_id`는 `hold`
* 하나의 thread는 여러 entry를 가질 수 있고, 프레임에 따라 entry가 바뀔 수 있음

---

## 3.4 Bi-directional Mentions (@)

### 텍스트 -> 3D

사용자가 메모 입력창에서 `@entity_alias`를 입력하면:

1. alias resolver가 `entity_id`를 찾는다.
2. 해당 3D 엔티티를 하이라이트한다.
3. 필요 시 `camera_view_id` 또는 엔티티 anchor 기준으로 카메라를 이동한다.
4. 메모 저장 시 `mentions[]`로 정규화 저장한다.

### 3D -> 텍스트

사용자가 3D 모델을 클릭/터치하면:

1. hit-test로 entity를 찾는다.
2. 확인 팝업을 띄운다.
3. “텍스트 메모” 또는 “음성 입력”을 선택하게 한다.
4. 선택된 entity를 기준으로 새 thread 또는 entry를 생성한다.

---

## 3.5 Annotation 표시 타입

MVP는 두 가지 표시 타입을 지원한다.

### Type A: 3D Anchored Pin

* 월드에 붙는 핀
* 모델과 함께 움직임

### Type B: Screen-fixed Callout

* 화면에 고정된 박스
* 3D anchor로 향하는 leader line을 그림
* 사용자가 요청한 기본 메모 타입은 **Type B**로 한다

> 기본값은 `Type B`로 두고, `Type A`는 옵션으로 제공한다.

---

## 3.6 3D 선택 후 입력 플로우

### 기본 플로우

1. 사용자가 3D 엔티티 클릭
2. 팝업: `이 구조에 메모를 추가할까요?`
3. 선택지:

   * 텍스트 메모
   * 음성 메모
   * 취소
4. 입력 후 thread 생성
5. 현재 프레임과 camera view를 기본값으로 저장

### 음성 입력 규칙

* 장치 권한 실패 시 즉시 텍스트 입력 fallback
* transcript 저장 전 사용자 확인 필요
* raw audio 저장은 옵션, 기본값은 비저장

## 4. Data Schema & Engineering

## 4.1 설계 원칙

### 원칙 1

**Authoring JSON과 Runtime JSON을 분리한다.**

### 원칙 2

**정적 지식과 동작 데이터를 분리한다.**

### 원칙 3

**World XYZ는 원본 저장값이 아니다.**
모든 위치는 `anchor_id` 또는 anchor 정의를 기준으로 저장한다.

### 원칙 4

**Action Instance는 Family를 복사하지 않는다.**
오직 `override_ops`만 가진다.

### 원칙 5

**Thread는 텍스트 본문과 시간축 상태를 분리한다.**

---

## 4.2 Multi-tier JSON Structure

### 1) Registry

정적 자산

포함:

* `entities`
* `groups`
* `anchors`
* `camera_views`
* `variant_profiles`
* `aliases`

### 2) Action Family

마스터 템플릿

포함:

* `frame_domain`
* `phases`
* `tracks`
* `distribution_rules`
* `proxy_profiles`
* `thread_template_ids`

### 3) Action Instance

개별 운동 차이점

포함:

* `family_id`
* `variant_profile_id`
* `override_ops`

### 4) Thread

사용자 작성/주석 데이터

포함:

* `target`
* `defaults`
* `entries`
* `display_track`
* `comments`

### 5) Resolved Runtime Clip

빌드 결과물

포함:

* family + instance override가 해소된 최종 데이터
* packed curves / texture refs
* runtime layer bindings

---

## 4.3 Action Family / Instance 오버라이드 규칙

### 허용 연산

* `replace_phase_range`
* `replace_track`
* `patch_track_keys`
* `remove_track`
* `set_proxy_param`
* `add_thread_ref`
* `remove_thread_template`
* `patch_thread_defaults`

### 의미

* 필드 없음: 상속
* `null`: 명시적 제거
* 빈 배열/객체: 의도적 empty override

### 런타임 금지

* 클라이언트에서 상속을 즉석 해석하지 않는다.
* 빌드 단계에서 반드시 `resolved_action_clip`을 생성한다.

---

## 4.4 Semantic Anchoring

모든 메모, 포인트, 콜아웃은 `anchor_id` 기반으로 저장한다.

### 권장 Anchor 타입

* `bone_landmark_lerp`
* `entity_surface_uv`
* `curve_param`
* `hybrid`

### 해결 순서

1. `anchor_id` 조회
2. canonical bind pose에서 위치 계산
3. variant retarget
4. posed transform 적용
5. `normal_offset_mm`만큼 띄워서 라벨/핀 위치 계산

### 금지

* raw `world_xyz`를 authoring truth로 저장
* triangle index / vertex index를 원본 ID처럼 사용

---

## 4.5 Build / Resolve Pipeline

### 빌드 파이프라인

1. `registry` 로드
2. `action.family` 로드
3. `action.instance` 로드
4. `override_ops` 순서대로 적용
5. thread template attach/remove 처리
6. distribution rule expand
7. proxy profile resolve
8. numeric curves pack
9. `resolved_action_clip.runtime.json` 생성

## 5.5 Timeline State Management

### 단일 소스 원칙

타임라인은 반드시 **single canonical store**를 가져야 한다.

런타임은 다음 3종 numeric cache를 중심으로 동작한다.

* `poseTex`
* `alphaTex`
* `proxyTex`

### 의미

* `poseTex`: bone/joint pose
* `alphaTex`: muscle activation/deform coeff
* `proxyTex`: ligament/nerve proxy intensity

### 금지

* authoring JSON에 runtime cache를 되쓰는 것
* UI state와 3D state를 분리된 시간축으로 운영하는 것

### MVP 기준

## 8. MVP Acceptance Criteria

### 기능 완료 조건

1. 7개 레이어가 개별 토글 가능해야 한다.
2. A/B/C authority 배지가 UI에 항상 드러나야 한다.
3. 타임라인 스크러빙 시 7개 레이어가 같은 프레임을 참조해야 한다.
4. 근육은 동적 변형, 인대/신경은 proxy tension, 경락/경혈/근막 경선은 static overlay로 동작해야 한다.
5. 사용자는 3D 엔티티 클릭 후 thread를 생성할 수 있어야 한다.
6. 사용자는 Global Memo와 Keyframed Memo를 둘 다 작성할 수 있어야 한다.
7. `@mention` 입력 시 해당 구조가 highlight되고 카메라가 이동해야 한다.
8. 공유 URL은 read-only 또는 clone 허용 옵션을 지원해야 한다.
9. 사용자 입력 장치 설정은 저장/복원되어야 한다.
10. authoring JSON과 runtime artifact는 물리적으로 분리된 파일/리소스로 관리되어야 한다.

### 품질 완료 조건

1. 스크러빙 중 레이어 desync가 없어야 한다.
2. note overlay가 프레임 기준으로 정확히 나타나고 사라져야 한다.
3. runtime에서 상속을 즉석 계산하지 않아야 한다.
4. raw world xyz가 authoring truth로 저장되지 않아야 한다.
5. C 레이어가 A/B 계산에 영향을 주지 않아야 한다.

## 9. Codex용 시스템 프롬프트 제약 5계명

1. **All authoring files are immutable normalized sources.**
   Never write resolved data back into `action.family.*.json`, `action.instance.*.json`, or `thread.*.json`. Resolve inheritance only into a separate runtime artifact.

2. **Use maps for registries and arrays only for ordered data.**
   `entities`, `anchors`, `camera_views`, `tracks`, and `entries` must be object maps keyed by stable IDs. `phases`, `keyframes`, `override_ops`, and `comments` must remain ordered arrays.

3. **Override semantics are strict.**
   Omitted field means “inherit”. `null` means “explicitly clear inherited value”. Empty array or empty object means “intentional override to empty”. Never invent fallback semantics.

4. **Timeline integrity is mandatory.**
   Every `frame_index` must be an integer inside the action `frame_domain`. Keyframes must be sorted ascending and unique per `(track_id, frame_index)`. Use `hold` interpolation for booleans and ID references, and numeric interpolation only for numeric channels.

5. **Anchors are the only source of truth for note positions.**
   A note must reference `anchor_id` or a valid entity-local anchor definition. Raw `world_xyz` may exist only in runtime cache and must never be treated as authoring truth. Hard-fail validation if any `target_entity_id`, `anchor_id`, `camera_view_id`, `group_id`, or `phase_id` cannot be resolved.
