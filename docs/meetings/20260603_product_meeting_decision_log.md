## 논의

- **제출 포지셔닝**: InBodyLIKE 지원 분야를 '라이프스타일 기술' 쪽으로 잡되, 사업계획서가 자유양식이므로 문제·해결책·시장·팀·인바디 협업 시나리오·PoC 계획을 자체 서사로 직접 설계한다.
- **인바디의 위치**: 인바디는 목적이 아니라 제출 맥락의 파트너이자 데이터 소스다. 인바디 수치는 진단/처방이 아니라 수업 관찰 참고 정보로 쓴다.
- **시연 방식**: 앱 화면만이 아니라 실제 수업 사용 흐름(녹음 → 전사 → 강사 피드백 → 리포트 발행)을 보여준다. 특정 녹음기/마이크를 강제하지 않고 블루투스 마이크·노트북·휴대폰·공간 배치는 현장 테스트 대상으로 둔다.
- **녹음 주체**: 녹음은 관리자/원장/강사 쪽에서 일어난다. 수련생의 무단 녹음은 부정 이용으로 보고, 수업 제공자의 권리·책임을 존중한다.
- **앱 이원화**: 제출용 웹앱은 한 흐름으로 보여주되, PoC 이후 관리자용 기록 앱과 수련생용 개인 아카이브 앱으로 나눈다. 관리자 앱은 회원권/매출 중심 CRM이 아니라 '정성적 데이터 관리'(수업 내용·수련생 반응·지도자 피드백·요가원 교재)에 가깝고, 데이터 소스는 음성·신체 스펙·성문화 자료다.
- **수련생 앱**: 단순 결과물 수신함이 아니라 소비자 매력의 핵심이다. 다만 기능을 덕지덕지 붙이지 않고 나이키 런처럼 한 가지 매력적인 기록/공유 경험을 먼저 찾는다.
- **시장 문제 정의**: 요가 시장의 문제는 '기록 계층의 부재'로 설명한다. 요가의 비정형성/블랙박스성은 결함이자 장점이며, 획일화 대신 각 요가원의 색과 지도자 스타일을 보존하는 적응형 아카이브로 설계한다.
- **현장 조사**: 수련생 인사이트는 설문(구글폼)보다 인터뷰/FGI(대화 녹음·분석)가 더 적합하다.
- **음성 API 파트너십**: 공동개발보다 셀프서브 API 검토가 현실적이다. 초기에는 API 문서 기반 자체 구현 가능성을 검토하고, PoC 규모·비용이 명확해진 뒤 협약/할인을 제안한다. 실시간 전사는 좋지만 필수는 아니며(데모는 실시간 느낌, 완제품은 수업 후 업로드/처리도 열어둠), 모바일 환경을 기본 전제로 둔다.
- **PoC 독립성**: 파트너가 없어도 PoC를 진행할 수 있어야 한다. 전사 엔진/CLOVA/Claude 등은 선택 가능한 기술 스택이고 핵심은 도메인 UX와 검수 구조다. 네이버 클라우드/오브젝트 스토리지 구조는 스폰서 핏(고객 소유 저장소·음성 처리·클라우드 인프라 검증)을 보여줄 수 있다.
- **BM 가정**: 하드웨어/온보딩 파견 모델(마이크·저장장치·설치 지원)은 매력적이지만 인건비·장비 조달·보험·재고·교육 비용까지 함께 모델링해야 한다. B2B(요가원 월 구독료)와 B2C(수련생 개인 아카이브 요금)는 결합해서 보고, 초기 매출 가정은 보수적으로 제시하되 업장당 월 매출·연 매출·100/1,000개 업장 확장 시나리오로 확장성을 보여준다.
- **도메인 톤**: 유동환의 바디카빙/요가철학이 제품 톤의 도메인 근거다. momso는 선생의 의도·수련생과의 관계·몸의 감각을 보존하는 도구여야 한다.

## 결정

- InBodyLIKE 지원 분야 = '라이프스타일 기술', 서사 구조는 자체 설계.
- 인바디 = 파트너/데이터 소스(진단·처방 아님, 수업 관찰 참고).
- 시연 = 실제 수업 사용 흐름(녹음→전사→피드백→리포트 발행), 녹음 장비 비강제.
- 녹음 권한 = 관리자/원장/강사. 수련생 무단 녹음은 부정 이용.
- 앱 이원화: PoC 이후 관리자 기록 앱 / 수련생 아카이브 앱 분리. 관리자 앱 = 정성적 데이터 관리.
- 시장 문제 = '기록 계층의 부재'. 적응형 아카이브로 요가원·지도자별 색 보존.
- 음성 API = 셀프서브 검토 우선, 실시간 전사는 비필수, 모바일 우선.
- 팀 비전 문장 = "사람과 사람의 소통을 안전하고, 진실되고, 지속가능하게 한다" → 제품 전체의 상위 문장. 요가 도메인은 시작점, 장기적으로 전문적 관계의 기록/공유 플랫폼으로 확장.
