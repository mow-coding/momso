# 2026-06-03 · 앱 기능 정의와 발렛파킹 BM 정리

> 한 줄 요약: momso를 "raw data → metadata → shareable data"를 만드는 관리자용 AI 작업대로 정의하고, 발렛파킹 비유로 "데이터 소유료가 아닌 아카이브 운영 자동화 비용"이라는 BM 명분을 확정했다.

- 참여: 유동환 · 김성균
- 맥락: 제출용 사업계획서와 데모를 다듬기 전에, momso가 "단순 녹음 앱·리포트 앱·CRM이 아니라 무엇인가"를 한 문장으로 정의하고, Web3.0 데이터 주권 철학이 약하게 만드는 BM을 보강할 비유·요금 구조를 합의하기 위한 회의.

## 논의

### momso의 한 문장 정의
- momso 관리자 앱은 사업자 인증을 마친 요가원 원장/강사가 수업을 녹음하고, 요가원 지식 소스와 과거 수업 기록을 바탕으로 AI 초안을 만들고, 강사가 검수한 기록만 수련생에게 발행하는 **AI 기록 작업대**다.
- 가장 중요한 화면은 녹음 버튼이 아니라 raw data와 metadata를 검수해 shareable data로 바꾸는 AI 작업대다.

### 데이터 세 계층
회의에서 가장 중요하게 정리된 구조. 이 세 계층이 UI/UX와 BM을 동시에 결정한다.

| 계층 | 정의 | 예시 | 권한/보관 원칙 |
|---|---|---|---|
| raw data | 아직 검수되지 않은 원자료 | 녹음 원본, 원전사본, 교재, Q&A, 수업 메모 | 가장 민감함. 기본 비공개. 고객 소유 저장소 우선 |
| metadata | raw data를 찾고 연결하기 위한 지도 | 태그, 주제, 수련생, 수업유형, RAG 색인, 연결 관계 | 다음 수업 AI 위키에 사용. 삭제/동의 철회 정책 필요 |
| shareable data | 강사가 검수한 공유 가능 자료 | 수련생 리포트, 공유 문장, API 응답용 요약 | 수련생 앱/API/SNS 확장의 기반 |

### 관리자 앱 탭 구조
- **홈** — 단순 대시보드가 아니라 RAG 필터의 시작점. 수업 유형·참여 수련생·동의 상태·수업 목적·이전 수업 맥락·인바디/웨어러블 지표를 고르는 순간 AI 위키가 참조 범위를 좁힌다.
- **녹음/수업 전 요청사항** — 녹음 시작 버튼만으로는 부족. 수업 전에 강사가 AI에게 기록 목적(수련생 리포트 중심, 다음 수업 관찰 포인트, 마케팅용 표현, 강사 말버릇 분석 등)을 미리 지정하는 전처리 프롬프트. 유동환은 사용자가 자기만의 기록 목적/`스킬`을 만드는 창이 필요하다고 제안.
- **AI 작업대/검수** — 핵심 탭. 전사본을 통째로 보여주지 않고 AI가 `대화의 초점`(호흡 패턴, 골반 안정성, 수련생 사적 고민, 강사 내부 메모, 용어 오류 가능성 등) 단위로 나눠주면, 강사는 초점별로 공유/내부/제외/보류를 결정. 강사는 문장 단위 편집자가 아니라 초점 검수자다.
- **공유/발행** — 검수된 shareable data만 내보냄. `리포트`는 화면 형태일 뿐, 본질은 shareable data. 같은 데이터가 텍스트 리포트·카카오톡·수련생 앱 카드·API 응답·외부 AI 입력으로 여러 형태를 가진다.

### AI 위키 / RAG
- 요가원을 오래 운영할수록 지식(녹음·전사·교재·철학 문서·Q&A·수련생 기록·민감 표현 등)이 커지므로, 매번 통째로 AI에 읽히면 느리고 비싸다. 지금 수업에 필요한 자료만 찾아 붙여주는 구조가 제품 언어로 `AI 위키`, 기술 언어로 `RAG`.
- metadata는 단순 태그가 아니라 AI가 자기 필터를 거는 `지도`. 강사 검수 결과가 다시 metadata로 쌓여 다음 수업의 검색·제안 정확도를 높인다.

### API의 의미
- 개발자 전용이 아니라 "momso에 축적된 내 데이터를 다른 앱에서 꺼내 쓰기 위한 약속". raw/metadata/shareable 포함 범위, 수련생/수업/기간 제한, 외부 AI 연결, 만료·철회를 선택 가능.
- 강사가 자기 ChatGPT·Gemini·Claude·Notion·Google Drive에서 자기 데이터를 쓸 수 있게 함. 단 모든 사용자에게 쉬운 기능은 아니라, 영업/온보딩 지원이 필요한 프리미엄 기능으로 본다.

### 발렛파킹 BM
- 완전한 Web3.0(데이터·연산 모두 고객 소유)만 고수하면 momso가 "앱 껍데기"처럼 보여 BM이 약해진다는 문제의식.
- 비유: 차(raw data)와 차고지(고객 소유 저장소)가 고객 것이어도 발렛파킹 회사는 돈을 번다. 직접 주차할 수 있지만 귀찮고 위험하고 시간이 들기 때문. momso도 녹음·업로드·요약·검수·발행·다음 수업 회수, 특히 개인정보·동의·민감 대화·용어·맥락을 모두 챙기는 운영 자동화 비용을 받는다.

| 비유 | momso 구조 |
|---|---|
| 차 | 녹음본, 전사본, 교재, Q&A, 수업 메모 같은 raw data |
| 차고지 | 요가원/원장 소유 저장소, 장기적으로 Naver Object Storage 같은 고객 소유 저장소 |
| 발렛파킹 기사 | Claude 등 AI 작업자 |
| 발렛 운영사 | momso |
| 주차 방식 | 요가원별 기록 문법·정리 방식·공유 기준 |
| 꺼내 쓰기 | 수련생 리포트, 다음 수업 참고, API, 외부 AI/SNS 활용 |

### Web2.5 하이브리드 요금 구조
제출용 메시지는 완전한 Web3.0이 아니라 Web2.5 하이브리드.

| 선택지 | 설명 | BM 의미 |
|---|---|---|
| momso 관리형 저장 | 초기에는 momso 서버/관리 클라우드에서 저장·처리 | 빠른 데모, 진입장벽 낮춤, 저장·처리 비용을 요금에 포함 |
| 고객 소유 저장소 연결 | Naver Object Storage 같은 고객 귀속 저장소 연결 | 데이터 주권, 스폰서 핏, 프리미엄/PoC 차별점 |
| BYOK | 고객 Claude/API key 연결 | AI 비용을 고객이 부담, momso는 오케스트레이션/운영 비용 수취 |
| API 프리미엄 | raw/metadata/shareable data를 외부 도구로 연결 | 고급 사용자/요가원 대상 확장 수익 |
| 온보딩/매니저 지원 | 저장소·마이크·녹음·API·리포트 스타일 세팅 | 스몰비즈니스 실사용 전환 비용 수취 |

- 중요 포인트: `이중 결제처럼 보이면 안 된다`. Naver Object Storage를 연결해도 "momso에도 내고 네이버에도 또 내나"로 느끼게 하면 안 되며, 기술 제휴/패키지형 요금처럼 복잡성을 숨겨 설명한다.

### AI 비용과 BYOK
- momso는 ChatGPT 자체를 만들려는 게 아니라, 채팅 화면 뒤의 raw data·metadata·RAG·전사/요약 처리가 핵심. 기본 모델은 Anthropic Claude 중심.
- Basic/Pro는 momso가 AI 크레딧을 포함해 제공, BYOK는 고객이 자기 Claude/API key를 연결하고 낮은 구독료 + momso는 운영/오케스트레이션 비용 수취.
- 주의: BYOK는 고객의 기존 Claude 대화 내역을 가져오는 게 아니라, 고객 key로 momso 작업대에서 AI 연산을 수행한다는 뜻.

### 수련생 앱의 위치
- 부속 리포트 뷰어가 아님. 두 연결 방식: ① 요가원 연결(빅블루 요가 같은 요가원의 철학·가격·수업 스타일 열람) ② 수업 리포트 연결(발행된 리포트를 날짜별 수신, 특정/여러/전체 아카이브 기반 AI 대화형 브랜치 세션).
- 장기적으로 개인 아카이브·프라이빗 피드·브랜치 세션·인포그래픽/명상음악/외부 공유 스킬로 확장. 단 사업계획서에서는 구현 완료가 아니라 PoC 확장 방향으로 표현.

## 결정
- momso의 애플리케이션 기능을 `raw data → metadata → shareable data`를 만드는 관리자용 AI 작업대로 정의한다.
- 데이터 세 계층(raw / metadata / shareable)을 UI·BM의 공통 뼈대로 채택한다.
- AI 위키/RAG는 방대한 요가원 지식 중 오늘 수업 관련 자료만 찾아오는 기억장치로 둔다.
- BM 명분은 "데이터 소유료가 아니라 고객 소유 데이터의 운영 자동화 비용"으로 확정한다(발렛파킹 비유).
- 제출 메시지는 완전한 Web3.0이 아니라 Web2.5 하이브리드로 간다.
- AI 비용은 기본 제공(Basic/Pro)과 BYOK 두 갈래로 구성한다.
- 데모는 화면 미관이 아니라 `raw → metadata → shareable` 구조를 직관적으로 보이게 하는 데 집중한다.

- API 프리미엄을 어느 사용자 등급부터 열지, 온보딩 지원을 어디까지 유료화할지.

### 사업계획서에서 조심할 표현

| 피해야 할 표현 | 더 안전한 표현 |
|---|---|
| Web3.0 SNS를 만든다 | Web2.5형 개인 아카이브와 공유 계층을 PoC에서 검증한다 |
| Naver Object Storage를 이미 연동했다 | 선발 후 Naver Object Storage 연결 구조를 검증한다 |
| Claude가 상주한다 | Claude 기반 AI 작업대 또는 BYOK 구조를 PoC에서 검증한다 |
| 인바디로 건강 상태를 분석한다 | 인바디 지표를 수업 관찰 참고 정보로 활용한다 |
| AI가 리포트를 발행한다 | AI는 초안을 제안하고 강사가 발행 범위를 확정한다 |
| 요가원 데이터를 우리 DB에 쌓는다 | 원본 데이터의 소유권을 존중하고, 운영/검수/발행 흐름을 제공한다 |
