# 2026-05-29 · Tiro 음성 처리 파트너십·PoC 협의

> 한 줄 요약: Tiro와 즉시 MOU를 맺는 자리가 아니라, momso가 구체적인 PoC 사용량·비용·데이터 조건·협력 대가를 문서로 정리해 다시 제안해야 한다는 점을 확인한 자리.

- 참여: 유동환 · 김성균 (상대측은 Tiro 담당자)
- 맥락: InBodyLIKE PoC에서 음성 녹음·전사·요약 레이어로 Tiro를 활용할 수 있는지 타진하기 위한 온보딩 겸 협력 타진 미팅. 미팅 전 대기 대화, 실제 Tiro 미팅, 미팅 후 내부 전략 피드백까지 세 갈래 논의를 한 흐름으로 정리.

## 논의

### 미팅 전 대화에서 끌어낸 제품 원칙
- Google Meet 초대 승인·로그인·2차 인증을 모르는 사용자가 많다는 사례 → 요가원 원장·강사·수련생 중 디지털 도구에 익숙하지 않은 사람이 많다는 전제. 녹음 시작·동의·리포트 확인·공유 절차는 극단적으로 단순해야 하고, 수업 직전 복잡한 로그인이 끼면 현장에서 바로 버려진다.
- AI 워크숍의 참가자 수준 격차 사례 → 기술 지식을 밀어 넣는 방식은 오래가지 못한다. momso는 원장·강사에게 "AI 전사/요약 기능"을 가르치는 제품이 아니라 "수업 후 기록·회원 케어·재방문·피드백 보존" 같은 현장 언어로 말해야 한다.
- 심리검사 플랫폼 마음사랑·밍글 사례 → 기능이 좋아도 도메인 종사자의 톤·속도·관계 형성 방식과 맞지 않으면 안 쓰인다. momso의 차별점은 기술 가능 여부가 아니라 "요가 도메인 사람들이 받아들일 수 있는 방식으로 설계했는가"에서 나온다.
- 요가 수련생 특성 → 체형 교정보다 마음의 자유·명상 효과·해방감을 기대하고 강사의 아우라·라포를 중시. 요가원은 몸만 다루는 공간이 아니라 내밀한 소통이 일어나는 공간. momso는 수업 중 오간 언어·감각·관계·피드백을 수련 기록으로 남긴다.
- FGI(Focus Group Interview) 기획 → 설문(Google Form)만으로는 한계. 진행자 유동환, 모더레이터 김성균, 기록 도구 Tiro. 산출물: 수련생 언어 사전, 리포트 니즈, 공유 거부감, 개인정보 민감도, 앱 톤앤매너. FGI 자체가 "대화 기반 기록화"의 작은 실험.

### 실제 Tiro 미팅
- Tiro가 먼저 확인한 것: (1) 실시간 기록이 필요한가, 수업 후 업로드로 충분한가 (2) 모바일인가 PC인가. momso는 "실시간 요약 필요 + 요가원에 데스크톱이 없어 모바일·태블릿 중심"이라 답했으나, 실시간 필수 여부는 내부 재검토 대상으로 남김.
- momso 설명: 단순 녹음·전사·요약 공유가 아니라 지도자-수련생 간 사적 기록을 안전하게 축적·공유하는 서비스. 데이터는 음성뿐 아니라 인바디 등 체성분 데이터까지 포함. 음성 부분에서 Tiro와 PoC가 가능한지 타진.
  - 잘한 점: 외주가 아니라 PoC 파트너 후보로 포지셔닝, 데이터가 음성에 그치지 않음을 명시.
  - 보완점: 기술 스펙이 아직 막연하다는 인상. 다음 제안은 "요가원 수·월 녹음 분·API·결과물·비용 조건"까지 숫자로.
- Tiro의 현재 입장: API 키 직접 발급으로 외부 개발자가 기능 연동 가능, MCP·CARI 제공, API 문서는 자체 검토 가능 수준으로 상세. 다만 외부 서비스와 공동개발 수준으로 붙인 전례는 없고 팀 규모가 작아 인력 파견·대외 협력은 부담. 그래도 국내 초기 창업팀과 좋은 관계에는 열려 있으며, 사용 규모·견적이 담긴 제안서를 받으면 내부 논의 가능.
- 협력 의향서·비용 할인 논의: momso는 MOU까지는 아니어도 협력 의향서 수준의 성문화된 근거, PoC 기간 API 비용 할인, 대가로 웰니스/요가 현장 사례 샷아웃·포트폴리오화를 제안. Tiro는 구체적 사용 정보·견적을 받은 뒤 내부 논의하겠다고 답하며, 초기 창업가 지원 의향은 있다고 함.
- Tiro가 보내주기로 한 것: API 개발자 문서, API 기반 유사 활용 사례(동물병원 상담 기록 축적 등 참고 케이스), API 질문의 내부 개발자 전달.
- 연희동 스튜디오 대면 미팅 제안 → Tiro는 즉답은 어렵지만 서울 팀원이 있고 요가를 하는 사람으로서 기회가 되면 방문하고 싶다는 긍정 뉘앙스(확정 아님).

### 미팅 후 내부 전략 피드백
- Tiro가 본 momso: 2인 초기팀(도메인 기획자 1 + 지원사업·문서·기술검토 1), 전담 개발자 없음, API 문서를 직접 검토해야 하는 단계, 공동개발 리소스를 투입하기엔 구체성 부족. "탑승하면 Tiro도 이득"이라는 논리보다 "PoC 기간 높은 API 사용량 + 성공 시 웰니스 사례·샷아웃 제공"이라는 대가 논리가 유효.
- 협상 전략 변화: Tiro가 적극 붙는다 → API를 제공하고 우리가 스스로 검토. 의향서를 바로 받는다 → 사용량·조건 제안 후 내부 검토 필요. Tiro 없으면 어렵다 → OpenAI API·자체 처리 등 대안 유지. 핵심: "Tiro와 협력하되, Tiro가 안 되면 멈추는 구조로 만들지 않는다."
- 관계 구분: Tiro = 더 성숙한 스타트업이자 협력 후보, 리소스 제공 의무 없음, 경우에 따라 갑에 가까움 → 사용량·비용·상호 이득·데이터 조건으로 접근. Bluepoint/InBody = 공모전·오픈이노베이션 운영 주체이자 평가자·지원자 → 팀 성장 가능성·문제 정의·PoC 설계·검증 계획으로 접근.
- 서비스 아키텍처 이원화(관리자용 / 소비자용):
  - 관리자용(원장·강사): 수업 예약·참석 확인, 녹음, 전사, 검수·편집, 공유. 녹음 권한 있음, 원본 접근 제한, B2B 과금 중심.
  - 소비자용(수련생·회원): 리포트 수신, 장기 수련 기록 보기, 공유/SNS. 녹음 권한 없음, 원본 접근 기본 불가, 초기 무료 또는 별도 검토.
  - 금지 원칙: 수련생이 요가원 허락 없이 수업을 녹음하는 구조는 만들지 않는다(지도자 노하우 침해·몰래 녹음 낙인·도입 거부·타인정보 노출 위험).
- 관리자 앱 4단계 흐름: (1) 수업 전 CRM성 정보 확인(수업 유형·개인/그룹/왓따 구분·예약 인원·방문 횟수·이전 기록 요약) → (2) 녹음(시작·실시간 전사 표시 가능성·상태·배터리/용량/네트워크 체크) → (3) 메모·편집(전사 확인·사적 내용 제거·노하우 노출 범위 조정·회원별 피드백 선별·이전 기록 연결) → (4) 공유(개인별 리포트 생성·지도자 검수 후 공유·원본 음성/전체 전사 기본 비공개·회원별 접근 권한 분리). → "지도자 검수형 수련 기록 시스템".
- 음성 처리 기술 선택지 비교: Tiro API(성능 검증·실시간/빠른 PoC ↔ 비용·벤더 의존·보관 조건), OpenAI API 직접(유연한 프롬프트 ↔ 파이프라인 직접 설계), 온디바이스 전사 후 텍스트 업로드(서버 비용 절감 ↔ 기기별 품질·장시간 안정성), 자체 서버 전사/요약(통제력·자산화 ↔ 초기 비용·시간), 스마트폰 기본 녹음 앱 + 후처리(MVP 빠름 ↔ 경험 통제 약함·수동 부담). 핵심 자산은 특정 전사 업체가 아니라 "동의-전사-검수-리포트-장기 아카이빙" 흐름.
- 데이터 주권·벤더 의존성: 음성 원본·전사문·요약문 모두 민감. 외부 API는 데이터가 외부 서버를 거침. 반드시 정리할 질문 — 원본 저장 위치, 전사/요약 보관 기간, 요청 시 삭제 가능 여부, 모델 학습 사용 여부, 로그 형태, 국외 서버 경유 여부, 개인정보 처리 위탁 계약 가능 여부.
- momso의 본질 = Voice Auto Archiving: 한 회의 요약(Tiro가 잘함)을 넘어, 회원별 수련 경험과 몸 데이터를 장기로 쌓아 다음 수업·리포트에 다시 쓰는 것. 자산 구조: 회원 → 수업 회차 → 음성 → 전사문 → 지도자 검수 메모 → 개인 리포트 → 인바디/체형/감각 데이터 → 장기 아카이브 → 다음 수업 참고. 따라서 Tiro는 핵심 후보일 수 있어도 본질 자체는 Tiro가 아니다.
- 도메인 집중·확장: 서비스는 요가에 집중하되 회사 비전은 요가에 갇히지 않음. 요가는 유동환이 가장 깊게 이해하고 현장 검증이 가능한 첫 도메인. 성공 시 필라테스·PT·재활운동·레슨형 시장으로 확장.
- 소비자 앱과 강제성 전략: 수련생이 먼저 "우리 요가원도 몸소 써주면 안 돼요?"라고 말하게 만들면 원장에게 자연스러운 도입 압력이 생김. 소비자 앱은 CRM처럼 보이면 안 되고, 인스타그램보다 사적·안전한 수련 기록 SNS에 가까워야 함. 기능 과다는 금물 — 처음엔 하나의 강한 경험만(오늘의 수련 카드, 감각 리포트, 선생님 한 문장 피드백, 나만 보는 기록, 제한적 공유 이미지 카드).

## 결정

- 이번 미팅의 실제 성과는 즉시 MOU가 아니라, momso가 "구체적 API 사용량·비용·데이터 조건·협력 대가"를 들고 다시 제안해야 한다는 확인.
- Tiro는 음성 처리 레이어의 후보 중 하나로 재정의(반드시 붙잡아야 할 필수 파트너가 아님). 대체 경로(OpenAI API·자체 처리)는 유지한다.
- 서비스는 관리자용/소비자용 두 층으로 (최소한 논리적으로) 분리한다.
- 수련생이 요가원 허락 없이 녹음하는 구조는 만들지 않는다(확정 금지 원칙).
- 강사 검수는 필수, 원본 음성·전체 전사본은 회원에게 기본 비공개.
- 제품의 핵심 자산은 특정 전사 업체가 아니라 "동의-전사-검수-리포트-장기 아카이빙" 흐름으로 둔다.
- 서비스는 요가 도메인에서 시작한다.

## 아직 미정

- 실시간 전사가 MVP 필수인지 (권장: 필수로 두지 말고 수업 후 안정 처리 + 지도자 검수 우선).
- 첫 PoC 수업 유형 (권장: 1:1 또는 2~4인 소그룹 우선).
- Tiro 협력 의향서/이메일 확인의 성문화 범위 (가능성은 열려 있으나 momso가 먼저 제안서를 만들어 보내야 함).
- PoC 기간 API 비용 할인·크레딧·스타트업 조건 가능 여부.
- 데이터 보관·삭제·학습 미사용 조건.
- 연희동 스튜디오 대면 미팅 (가능성만 열린 상태, 제안서 전달 후 자연스럽게 추진).
