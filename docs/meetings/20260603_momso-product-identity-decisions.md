# 2026-06-03 · momso 제품 정체성과 데이터 구조 결정

> 한 줄 요약: momso를 "CRM/녹음 앱"이 아니라 "강사 검수 기반 기록·검수·공유 계층"으로 정의하고, 데이터를 raw/metadata/shareable 3종으로 나누며 Web2.5·BYOK를 사업 서사에 어떻게 배치할지 합의했다.

- 참여: 유동환 · 김성균
- 맥락: 여러 차례의 전화회의에서 흩어져 나온 momso의 제품·사업·아키텍처 결정을 한데 모아, 지원사업 제출 서사와 PoC 검증 범위를 정리하기 위함. 김성균은 설계·기술 구조·BM·서사를, 유동환은 요가원 운영자로서 도메인 지식과 수련생/지도자 관계 맥락을 제공한다.

## 논의

- **제품 정체성**: momso를 단순 CRM(회원관리/매출관리)으로 정의하지 않는다. 제품의 첫 문장은 `수업 기록`·`강사 검수`·`수련생 공유`를 중심에 둔다. momso는 음성 녹음 앱이 아니라 기록/검수/공유 계층이며, STT(Tiro·CLOVA 등)는 부품이고 차별점은 도메인 하네스(AI harness)와 공유 범위 결정이다.
- **외부 기술 위치**: Tiro·InBody·CLOVA·Claude는 momso의 수족 또는 파트너 기술이지 제품 정체성이 아니다. 사업계획서에서 외부 기술 연동을 "구현 완료"처럼 과장하지 않고 PoC 검증 방향으로 둔다.
- **앱 분리**: 관리자 앱과 수련생 앱은 분리되는 방향이 맞다. 제출용 웹앱에서는 한 흐름으로 보여주되, PoC 이후 별도 제품면으로 설계한다. 수련생 앱은 단순 리포트 뷰어가 아니라 개인 아카이브/피드/API/SNS 확장의 씨앗이다(유동환 문제제기, 김성균 구조화).
- **데이터 3종 분리**: raw data(원본 녹음·원전사본·미검수 초안), metadata(수업 날짜·수련생·주제·요가 용어·태그·연결 관계), shareable data(강사가 발행 확정한 기록)로 나눠 저장·권한관리한다.
- **Web2.5 / 데이터 주권**: Web3.0을 전면 주장하지 않고 Web2.5 하이브리드로 말한다(중앙 서버 + 사용자 소유 저장소 병행, 자기 저장소 전환은 PoC 확장). 데이터 주권은 제품 첫 문장이 아니라 신뢰 설계의 근거로 배치한다.
- **BM 비유**: momso는 사용자 데이터를 "소유"하는 회사가 아니라 발렛처럼 기록 운영을 대행하는 회사다. 수익은 데이터 소유권이 아니라 운영 자동화·검수 흐름·저장/AI 연결·프리미엄 관리에서 나온다. 또한 특정 메모법을 강제하지 않고, 요가원/지도자/수련생별 표현 방식을 끌어내는 적응형 아카이빙을 지향한다(유동환 감각 제시, 김성균 구조화).
- **저장·AI**: Naver Object Storage는 "사용자 소유 차고지" 가설로 둔다(네이버 로그인/오브젝트 스토리지/CLOVA 연동은 선발 후 PoC에서 검증). 기본 AI는 Claude 중심으로 설명하되 실제 모델 연동은 PoC 검증 방향으로 둔다.
- **BYOK 과금**: BYOK(Bring Your Own Key)는 철학과 BM을 동시에 설명하는 장치다. Basic/Pro는 momso가 AI 비용을 포함하고, Sovereign/BYOK는 고객이 자기 key 비용을 부담하며 momso는 오케스트레이션 비용을 받는다.

## 결정

1. momso = 강사 검수 기반 "기록·검수·공유 계층" (CRM/녹음 앱 아님). 첫 문장은 수업 기록·강사 검수·수련생 공유 중심.
2. HITL 원칙 채택: AI는 초안만 제안하고, 민감정보·요가 용어·공유 가능 여부는 강사가 최종 확정한다.
3. Tiro/InBody/CLOVA/Claude는 파트너/부품 기술로 위치시키고, 연동은 "PoC 검증" 표현을 쓴다(구현 완료로 과장 금지).
4. 관리자 앱 / 수련생 앱은 장기적으로 분리. 수련생 앱은 개인 아카이브·피드·API·SNS 확장의 씨앗으로 설계한다.
5. 데이터를 raw data / metadata / shareable data 3종으로 분리해 저장·권한관리한다.
6. 서사는 Web2.5 하이브리드로 말하고, 데이터 주권은 신뢰 설계 근거로 배치(첫 문장에 두지 않음).
7. BM은 "발렛형 기록 운영 대행" + 적응형 아카이빙으로 정의한다.
8. 과금: Basic/Pro = AI 비용 포함, Sovereign/BYOK = 고객이 key 비용 부담 + momso는 오케스트레이션 비용.
9. 유동환(도메인·수련·관계 철학)과 김성균(데이터 주권·설계·BM 철학)의 철학을 같은 위계로 보존한다.

### 제품 구조로 번역한 합의

**관리자용 momso**
- 수업 전: 오늘 수업·참여자·동의 상태·요청사항/프롬프트 확인.
- 수업 중: 원장/강사 계정에서만 녹음 시작.
- 수업 후: AI가 전사/요약/공유 후보 제안.
- 검수: 강사가 요가 용어·민감구간·공유 가능 여부 확정.
- 발행: 검수된 기록만 수련생에게 공유.

**수련생용 momso**
- 강사가 발행한 리포트만 본다(원본 녹음은 기본 비공개).
- 자기 수련 기록은 개인 아카이브로 누적.
- 외부 API·SNS·다른 AI 엔진 연결은 동의 후 확장.
- 장기적으로 리포트 기반 브랜치 세션·개인 피드·프라이빗 SNS로 확장 가능.

**데이터/AI 계층**
- raw data: 원본 녹음·원전사본·미검수 초안.
- metadata: 수업 날짜·수련생·주제·요가 용어·태그·연결 관계.
- shareable data: 강사가 발행 확정한 기록.
- AI workbench: Claude 또는 다른 LLM이 초안을 제안하는 작업대.
- BYOK: 고객이 자기 AI key를 꽂는 선택지.
- user-owned storage: Naver Object Storage 같은 사용자 소유 저장소 가설.

### 사업계획서에 넣을 수 있는 문장

> momso는 요가 수업에서 사라지는 지도자의 말과 수련생의 감각을, 강사가 검수한 개인 기록으로 바꾸는 모바일 웹앱입니다. AI는 초안을 제안하지만 공유 범위는 강사가 확정하고, 원본 녹음은 기본 비공개로 둡니다.
>
> 선발 후 PoC에서는 Naver Login/Object Storage/CLOVA Speech와 Claude/BYOK 구조를 검증해, 요가원과 수련생이 자기 기록을 잃지 않으면서도 AI 기반 아카이빙을 사용할 수 있는 Web2.5형 기록 운영 모델을 실험합니다.
