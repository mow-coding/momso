# 2026-06-03 · momso 제품 철학·데이터 계층·Web2.5 BM 정렬

> 한 줄 요약: momso를 단순 녹음/요약 앱이 아니라, 사용자가 자기 기록·저장소를 소유한 채 AI 기록 자동화를 쓰는 Web2.5 기록 운영 시스템으로 정의하고, 그 위에서 관리자/수련생 앱 구조·데이터 3계층·AI 프로바이더/BYOK 전략을 합의했다.

- 참여: 유동환 · 김성균
- 맥락: InBodyLIKE 제출용 PRD·프로토타입·사업계획서에 반영할 제품 철학과 BM 방향을 같은 날 세 차례 통화(누적 약 3시간 47분)로 정렬했다. 본 회의록은 그 핵심만 추린 것이다.

## 논의

### 큰 방향 (세 통화 공통)

**1. momso는 메모법을 강요하는 앱이 아니다.** 사람·요가원마다 말하고 기억하고 가르치고 기록하는 방식이 다르므로, momso가 표준 메모법/리포트 양식/아카이빙 체계를 강요하면 안 된다. AI는 "이렇게 기록하라"고 가르치는 도구가 아니라, 사용자가 이미 가진 기록·수업 방식을 발견해 정리해 주는 도구여야 한다.

| 잘못된 방향 | 맞는 방향 |
|---|---|
| momso식 기록법을 강요한다 | 각 요가원의 기록 문법을 발견한다 |
| 모든 리포트를 같은 템플릿으로 만든다 | 수업/강사/수련생/철학에 따라 형식을 적응시킨다 |
| AI가 정답을 말한다 | AI가 초안을 제안하고 사람이 확정한다 |
| 데이터가 momso 안에 갇힌다 | 데이터는 사용자가 소유하고 momso는 운영을 돕는다 |

**2. 두 사람의 철학은 같은 위계로 반영된다.**

| 제공자 | 핵심 철학 | momso에서의 역할 |
|---|---|---|
| 유동환 | 사람 사이 소통을 안전·진실·지속가능하게 만든다 | 요가/웰니스 도메인의 의미, 수업의 암묵지, 지도자-수련생 관계의 윤리 |
| 김성균 | 데이터 주권과 Web2.5 구조로 사용자가 자기 기록을 잃지 않게 한다 | 저장소, API, BYOK, 개인 아카이브, 벤더 의존성 감소 |

통합 문장: "momso는 사람들이 의미 있는 기록을 축적하고 생성하는 삶을 살 수 있도록 돕는다." (단, 공모전 제출 문서에서는 내부/사이드프로젝트 용어를 전면에 쓰지 않고 momso 언어로 번역해 설명한다.)

**3. 데이터는 세 계층으로 나눈다.**

| 계층 | 예시 | 소유/보관 원칙 | 제품에서 하는 일 |
|---|---|---|---|
| Raw data | 녹음 원본, 전사 원문, 교재 PDF, 강사 메모, Q&A | 원칙적으로 요가원/수련생 데이터 | 수집, 보관 위치 연결, 권한/동의 관리 |
| Metadata | 요가원 철학, 수업 주제, 반복 지도 표현, 수련생 상태 변화, 검색/RAG 인덱스 | 파생물이므로 사용자 통제 필요 | AI가 다음 수업에서 참고할 지식 지도 생성 |
| Shareable data | 강사가 검수한 리포트, 공유 가능 문장, 외부 API 내보낼 요약 | 공유 범위를 사람이 확정 | 수련생 앱, 링크, API, 콘텐츠 확장 |

**4. 완제품 데이터/AI 흐름.** 사업자 인증된 원장이 수업 생성 → 원장/권한 강사가 녹음 → 원본은 정책대로 보관(장기 방향은 고객 소유 저장소 우선) → 백엔드가 AI 전사/요약 파이프라인으로 전달 → AI가 교재·이전 기록·강사 스타일·동의 범위 등 지식 소스 참고 → AI가 초안/요약/공유·제외 후보/다음 수업 참고 포인트 제안 → 강사 검수·확정 → 확정 기록만 수련생 앱/링크/API로 발행 → 발행·검수 결과가 다음 수업 지식 소스로 환류. 핵심: AI는 발행하지 않고 제안만, 강사가 확정. 원본 녹음은 기본 비공개. 다음 수업 AI의 기억도 사용자의 통제·삭제·내보내기 권한 아래 있어야 한다.

### 통화 1: 개인 아카이브·BM·철학 출발점

- momso는 "기록의 중요성을 아는 사람의 귀찮음"(매번 녹음·정리·분류·재활용)을 대신 처리하는 운영 서비스다. BM은 이 운영 비용 대행에서 나온다.
- Web3.0 철학만으로는 BM이 약해진다(투자자: "그럼 어디서 버나"). 기본형은 서버/클라우드 운영형, 주권을 원하는 사용자는 자기 저장소/BYOK로 전환하는 하이브리드가 필요 → Web2.5.
- 발렛파킹 비유: 차·차고지는 고객 것이지만 직접 주차할 시간이 아깝다. 발렛 업체는 차를 소유하지 않아도 돈을 번다. momso도 데이터·저장소를 소유하지 않고 기록 운영 자동화·AI 정리 대행으로 수익.
- SOA 관점: 음성 기록을 쌓는 서비스, 쌓인 걸 읽어 답하는 LLM은 있으나, 사용자 세계관에 맞게 계속 정리·재조직하는 서비스는 부족 → momso가 노릴 빈자리.

### 통화 2: 관리자 앱/수련생 앱 UI·데이터 계층·API BM

관리자 앱 최소 흐름:

| 탭/화면 | 역할 | 중요 UX |
|---|---|---|
| 홈 | 수업·수련생 선택, CRM 일부 | 수업 유형·참여자 선택 시 관련 지식 소스 자동 필터링 |
| 녹음 | 수업 녹음 시작 | 녹음 전 "오늘 무엇을 중심으로 기록할지" 요청사항 입력 |
| AI 초안/검수 | 전사/요약/공유 후보 검수 | AI 초안 제안, 강사 확정 |
| 초점 정리 | 대화 초점 시각화 | 워드클라우드/마인드맵/블록 텍스트 중 검토 |
| 공유 | 검수된 기록 발행 | shareable data만 수련생 공개 |

- 녹음 전 요청사항이 중요: "개인 지도 리포트 중심", "마케팅 자료 겸용", "특정 수련생 몸 감각 변화 중심" 등 방향을 미리 주면 AI 정확도가 오른다.
- 검수 화면은 "요약문 수정"이 아니라 강사가 공유 가능/내부 참고/민감 제외/용어 확인 필요/리포트에서 빼기/다음 수업에 남기기 단위로 조절하는 화면이어야 한다. AI가 먼저 초점·블록·후보를 나눠주고 강사는 선택·제외만.
- 수련생 앱: 단순 PDF 뷰어가 아니라 요가원 검색/링크 연결 → 철학·스타일·가격·안내 열람 → 날짜별 리포트 수신 → 리포트 1개/여러 개/전체를 골라 AI와 대화(브랜치 세션) → 장기적으로 인포그래픽·이미지·명상 음악·콘텐츠 플레이그라운드.
- 제품 범위 한 문장: Raw data 수집 → metadata 산출 → shareable data 생성 → 공유 환경 제공. 이후 릴스/유튜브/책/홍보물로의 재창조는 사용자 영역이되 momso가 API·수련생 앱으로 길을 연다.
- API BM 후보: 요가원 내부 시스템 연동, 외부 AI 도구(ChatGPT/Gemini/Claude) 연결, 수련생 개인 아카이브 내보내기, SNS/콘텐츠 도구 연동, 프리미엄 요가원 대상 데이터 포터빌리티/백업.

### 통화 3: AI 프로바이더·BYOK·저장소·마케팅 위계·채용

- AI 프로바이더 전략: 초기에는 다중 모델 동시 지원을 피한다(모델별 프롬프트/RAG/검색 튜닝 비용·복잡도). 스폰서 맥락상 Anthropic Claude를 기본 모델로 제시하는 것이 자연스럽고, Claude는 인간적 맥락 해석·정성 정리에 강점.

| 플랜 | AI 비용/인증 방식 | 고객 경험 | momso 수익 |
|---|---|---|---|
| Basic | momso가 Claude/API 비용 부담, 사용량 제한 | 그냥 쓴다 | 구독료 + 사용량 제한 |
| Pro | AI 크레딧/사용량 더 제공 | 많이 쓰는 요가원 | 구독료 + 초과 사용량 |
| Sovereign/BYOK | 고객이 자기 Claude API key/계정 연결 | 데이터·비용 통제권 확대 | 낮은 구독료 + 운영/오케스트레이션 비용 |

- BYOK는 차별점이지만 너무 일찍 전면 기능으로 약속하면 구현 부담이 크다 → 제출 문서에서는 "선발 후 PoC에서 검증할 확장 방향"으로 표현.
- 저장소 핵심 구조: momso 서버가 모든 녹음본을 영구 보관하는 게 아니라, 고객 소유 저장소를 연결하고 momso가 그 위에서 AI 기록 운영을 수행. 단계: ① 초기 데모는 더미·로컬/프론트 중심 ② PoC 초기는 momso 관리 클라우드 + 강한 권한/삭제 정책 ③ PoC 확장은 원장 계정 연결 Naver Object Storage 버킷 ④ 장기는 고객별 저장소 선택권·BYOC/객체 스토리지·데이터 내보내기.
- 마케팅 메시지 위계: ① 전면 = 요가 도메인의 안전·진실·지속가능 소통 ② 제품 = 강사 검수 기록만 수련생에 공유 ③ 기술 = AI 제안, 사람이 공유 범위 확정 ④ 장기 철학 = 데이터는 사용자 저장소·권한 아래 ⑤ PoC 확장 = Naver Object Storage·Claude·BYOK·API/SNS. 발표를 "우리는 Web3.0 스타트업"으로 시작하면 안 되고, 먼저 "수업에서 사라지는 말과 감각" 문제를 보여준 뒤 저장/AI 구조 필요성을 설명한다.

## 결정

- momso의 핵심 비용 항목은 "AI 요약 비용"이 아니라 "기록 운영 자동화 비용"으로 잡는다.
- 데이터 3계층(Raw / Metadata / Shareable)을 제품 데이터 모델의 기본 골격으로 채택한다.
- AI는 제안만, 강사가 검수·확정, 확정 기록만 발행한다(원본 녹음 기본 비공개).
- 기본 AI 모델은 Anthropic Claude로 제시하고, 초기에는 다중 모델 동시 지원을 하지 않는다.
- 저장소 장기 방향은 고객 소유 저장소(Naver Object Storage 등) 연결형 Web2.5 구조로 한다.
- 데이터 주권은 첫 문장으로 내세우지 않는다. 위계상 "안전한 수업 기록·공유"를 전면에 두고 주권/BYOK는 뒤에 배치한다.
- 제출 문서에서 Web3.0 용어와 내부 사이드프로젝트 용어를 전면에 쓰지 않고 momso 언어로 번역한다.
